# Security

> Security model, authentication flow, data encryption, and hardening measures.

## Security Model Overview

```
┌─────────────────────────────────────────────────────────┐
│                    SECURITY LAYERS                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. Traefik ──── TLS termination (HTTPS)               │
│                  Automatic cert renewal                  │
│                  HTTP → HTTPS redirect                   │
│                                                         │
│  2. Middleware ── JWT verification                       │
│                  Route protection (/panel/*)             │
│                                                         │
│  3. API Layer ── Input validation                       │
│                  Error sanitization                      │
│                                                         │
│  4. Auth ─────── bcrypt password hashing (12 rounds)    │
│                  JWT HS256 (HttpOnly cookie)             │
│                  7-day session TTL                       │
│                                                         │
│  5. Data ─────── AES-256-GCM encryption (credentials)  │
│                  Encryption key from env var             │
│                                                         │
│  6. Network ──── PostgreSQL on private bridge            │
│                  No external port exposure               │
│                  Container isolation                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## Authentication

### Login Flow

```
Client                    Server
  │                         │
  ├── POST /api/auth/login ─┤
  │   { username, password } │
  │                         ├── Find user in DB
  │                         ├── bcrypt.compare(password, hash)
  │                         ├── Create JWT token (jose, HS256)
  │                         ├── Set HttpOnly cookie
  │                         │
  ◄── 200 { success, data } ┤
  │                         │
  ├── GET /api/stats ───────┤
  │   Cookie: vps-session    │
  │                         ├── middleware.ts verifies JWT
  │                         ├── Verify token → extract userId
  │                         │
  ◄── 200 { data }          │
```

### Password Hashing

- **Library:** bcryptjs
- **Rounds:** 12 (secure without being too slow)
- **Storage:** Only the hash is stored — NEVER the plaintext

```typescript
// Hash when creating a user
const hash = await bcrypt.hash(password, 12);

// Verify when logging in
const valid = await bcrypt.compare(inputPassword, storedHash);
```

### JWT Token

- **Library:** jose (lighter than jsonwebtoken, uses native crypto)
- **Algorithm:** HS256 (HMAC + SHA-256)
- **Secret:** `JWT_SECRET` (64 hex chars, generated by `deploy.sh`)
- **Payload:** `{ userId, username, iat, exp }`
- **TTL:** 7 days

### Cookie Configuration

```typescript
{
  name: "vps-session",
  httpOnly: true,        // JS cannot read it → XSS protection
  secure: production,    // Only sent over HTTPS in production
  sameSite: "lax",       // Basic CSRF protection
  maxAge: 7 * 24 * 3600, // 7 days
  path: "/"
}
```

### Middleware Protection

`src/middleware.ts` protects all routes in the `(panel)` group:

- Checks that the `vps-session` cookie exists
- Verifies the JWT token
- If invalid → redirects to `/login`
- API routes (`/api/*`) return 401 instead of redirecting

## Data Encryption

### Why encrypt?

SSH credentials (passwords, private keys) are stored in PostgreSQL. If the database is compromised, attackers would gain full SSH access to every managed server. AES-256-GCM encryption prevents this.

### Algorithm: AES-256-GCM

| Property        | Value                              |
| --------------- | ---------------------------------- |
| Algorithm       | AES-256-GCM                        |
| Key size        | 256 bits (32 bytes)                |
| IV size         | 128 bits (16 bytes, random each time) |
| Auth Tag size   | 128 bits (16 bytes)                |
| Output format   | `base64(IV + ciphertext + authTag)` |

### Why GCM?

- **Authenticated:** Automatically detects if ciphertext has been tampered with
- **Fast:** Uses AES-NI hardware acceleration
- **Standard:** NIST approved, widely used in TLS 1.3

### Encryption Key

- Environment variable: `ENCRYPTION_KEY`
- Requirement: Exactly 64 hex characters (= 32 bytes)
- Generated by: `openssl rand -hex 32` (inside `deploy.sh`)
- **CRITICAL:** If this key is lost, ALL credentials in the DB become unrecoverable

### Code Flow

```typescript
// Encrypt when saving a server
import { encrypt, decrypt } from '@/lib/crypto';

const encryptedPass = encrypt(sshPassword);
// → "SGVsbG8gV29ybGQ=" (base64)

await prisma.server.create({
  data: { ...rest, encryptedPass }
});

// Decrypt when connecting via SSH
const server = await prisma.server.findUnique({ where: { id } });
const password = decrypt(server.encryptedPass);
// → "original-ssh-password"
```

## SSH Security

### Auto-Accept Host Keys

`ssh2-promise` accepts all host keys by default when no `hostVerifier` callback is provided. This is an intentional design decision:

- **Reason:** Automation — no manual confirmation needed when adding new servers
- **Risk:** Vulnerable to Man-in-the-Middle attacks on untrusted networks
- **Mitigation:** Use within trusted server-to-server environments

### Supported Algorithms

```typescript
serverHostKey: [
  'ssh-ed25519',
  'ecdsa-sha2-nistp256',
  'ecdsa-sha2-nistp384',
  'ecdsa-sha2-nistp521',
  'rsa-sha2-512',
  'rsa-sha2-256',
  'ssh-rsa'
]
```

### Connection Timeout

- **Timeout:** 10 seconds
- If the server doesn't respond within 10s → throws an error

## Network Security

### Container Isolation

```
Internet → Traefik → [traefik_network] → App → [internal] → PostgreSQL
                                                    │
                                           ┌────────┘
                                           │
                              PostgreSQL has NO path
                              to the internet (internal network only)
```

### No Published Ports

- App container: No published ports (Traefik reaches it via Docker network)
- DB container: No published ports (only the app reaches it via internal network)

### TLS/HTTPS

- Traefik handles TLS termination
- Automatically provisions and renews Let's Encrypt certificates
- HTTP automatically redirects to HTTPS

## Secret Management

### Secrets in the System

| Secret            | Purpose                           | Generated by       | Length       |
| ----------------- | --------------------------------- | ------------------ | ------------ |
| `DB_PASSWORD`     | PostgreSQL authentication         | `deploy.sh`        | 48 hex chars |
| `JWT_SECRET`      | Signing JWT tokens                | `deploy.sh`        | 64 hex chars |
| `ENCRYPTION_KEY`  | AES-256-GCM encryption            | `deploy.sh`        | 64 hex chars |
| `ADMIN_PASSWORD`  | Initial admin account             | User input         | User-defined |

### Principles

1. **Never hardcode** secrets in source code
2. **Never commit** `.env` to git (excluded in `.gitignore`)
3. **Generate randomly** with `openssl rand -hex` (cryptographically secure)
4. **Never log** secrets to console or API responses

## Current Limitations

| Limitation                      | Severity | Potential Improvement                   |
| ------------------------------- | -------- | --------------------------------------- |
| No rate limiting                | Medium   | Add rate limiter middleware              |
| No 2FA                          | Medium   | Add TOTP (Google Authenticator)         |
| No audit log                    | Low      | Log all critical actions to the DB      |
| SSH host keys not verified      | Low      | Store known_hosts and verify            |
| No explicit CORS policy         | Low      | Add CORS headers to API routes          |
| No CSP headers                  | Low      | Add Content-Security-Policy headers     |

## Deployment Security Checklist

- [ ] Domain is pointed to the server
- [ ] Traefik is running with TLS enabled
- [ ] `.env` is NOT committed to git
- [ ] `ENCRYPTION_KEY` is backed up securely (losing it = losing all credentials)
- [ ] Admin password is strong (12+ chars, mixed case, numbers, symbols)
- [ ] Firewall only allows ports 80 and 443 (for Traefik); all others closed
- [ ] SSH key-based auth is used for the host server itself (no password SSH to host)
